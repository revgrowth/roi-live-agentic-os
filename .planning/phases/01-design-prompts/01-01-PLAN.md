<task type="auto">
  <name>Task 2: Create design language document with shared tokens</name>
  <files>
    .planning/phases/01-design-prompts/design-language.md
  </files>
  <action>
  Using the research from Task 1 (both files), synthesise a design language document that defines the shared visual foundation for the Command Centre.

  The document must define:

  **Colour palette:**
  - Primary, secondary, accent colours (hex values)
  - Background colours (page, card, sidebar)
  - Status colours: queued (neutral), running (active/blue), review (amber/warning), done (green/success), error (red)
  - Text colours: primary, secondary, muted
  - Border and divider colours
  - Derive from Vibe Kanban's clean light aesthetic -- not dark mode, not overly colourful

  **Typography:**
  - Font family (system or specific -- e.g., Inter)
  - Size scale: page title, section heading, card title, body text, caption/meta
  - Font weights for each level
  - Line heights

  **Spacing:**
  - Base unit and scale (e.g., 4px base: 4, 8, 12, 16, 24, 32, 48)
  - Page margins, card padding, column gaps
  - Section spacing

  **Component styles:**
  - Card: border radius, shadow, padding, hover state
  - Button: primary, secondary, ghost -- sizing, border radius, padding
  - Input: text field, textarea, select -- border, focus state
  - Badge/tag: status badges, level badges (Task/Project/GSD)
  - Stats bar: layout, number display, label styling
  - Slide-out panel: width, overlay, animation direction
  - Nav bar: height, item spacing, active indicator
  - Column: header style, drop zone indicator

  **States:**
  - Empty state: icon + message pattern
  - Loading state: skeleton or spinner pattern
  - Error state: colour, icon, message pattern
  - Hover/focus/active states for interactive elements

  **Design principles** (3-5 bullet points):
  - e.g., "Clean and minimal -- no decorative elements, every element serves a function"
  - e.g., "Status communicated through colour, not icons or text alone"
  - Derived from reference analysis

  Format as a well-structured markdown document that Plan 01-02 can directly reference when crafting individual view prompts. Use a consistent format (e.g., token tables) so prompts can say "use the card component style from the design language" unambiguously.
  </action>
  <verify>
  `design-language.md` exists and contains:
  - Colour palette with hex values (at least 15 colour tokens)
  - Typography scale with specific sizes and weights
  - Spacing scale with numeric values
  - Component styles for at least: card, button, input, badge, nav bar, stats bar
  - State patterns for empty, loading, error
  - Design principles section
  </verify>
  <done>Design language document defines all shared tokens with specific values. Any prompt writer can reference it to produce visually consistent Stitch prompts across all views.</done>
